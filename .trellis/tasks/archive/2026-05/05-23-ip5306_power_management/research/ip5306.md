## Relevant Facts

- The project netlist identifies U2 as `IP5306-I2C`.
- U2 pin mapping from the project netlist:
  - VIN -> `VBUS`.
  - LED1/SCL -> `IP5306T_I2C_SCK`.
  - LED2/SDA -> `IP5306T_I2C_SDA`.
  - LED3 -> not connected.
  - KEY -> IP5306 wakeup/control node.
  - BAT -> battery side through R7.
  - SW -> external 1 uH inductor.
  - VOUT -> switch path into `SYS_POWER`.
  - EP -> GND.
- Vendor PDF describes ESOP8 pins VIN, LED1, LED2, LED3, KEY, BAT, SW, VOUT, and PowerPAD/GND.
- The same PDF states:
  - BAT operating range is 3.0-4.4 V.
  - Typical charge current is 2.1 A.
  - Boost output current capability is 2.4 A.
  - Light-load auto-detection is approximately current <45 mA for 32 s.
  - A key press longer than 30 ms and shorter than 2 s is a short press that wakes/turns on output.
- The user requirement asks for an active-low P0.22 keepalive pulse every 8 s, width 300-500 ms. That fits the key timing window and stays well below the documented 2 s long-press range.

## Register / I2C Risk

- Common examples use IP5306 I2C address `0x75`, but the project netlist cannot encode I2C address.
- Available public register maps are not consistently primary vendor sources.
- First firmware version should:
  - Define address in devicetree so it is easy to change.
  - Probe with a harmless read if possible.
  - Provide raw read/write helper functions.
  - Avoid writing unknown registers during boot.
  - Avoid high-level charging/full/boost status helpers unless a verified register definition is added.

## Implementation Consequences

- Keep the IP5306 adapter conservative: probe/log only, no configuration writes.
- Put light-load mitigation primarily in the KEY keepalive pulse, not in register programming.
- Battery state should come from external ADC measurement first, not IP5306 LED/I2C state decoding.
