# Host Configuration GUI Contracts

## Scenario: PySide6 desktop configuration tool

### 1. Scope / Trigger

- Trigger: code under `pc_gui/` adds or changes device communication, frame/protobuf handling, serial transport, configuration models, or PySide6 UI pages.
- Goal: keep UI widgets independent from protocol bytes and serial I/O while preserving developer logs for every request.

### 2. Signatures

- `FrameCodec.encode(payload: bytes) -> bytes`
- `FrameCodec.decode(frame: bytes) -> bytes`
- `FrameStreamDecoder.feed(data: bytes) -> list[bytes]`
- `DeviceConfig.to_proto() -> device_comm_pb2.DeviceConfig`
- `DeviceConfig.from_proto(proto) -> DeviceConfig`
- `DeviceProtocol.request(message: device_comm_pb2.DeviceMessage) -> device_comm_pb2.DeviceMessage`
- `DeviceClient.connect() -> None`
- `DeviceClient.hello(app_name: str = "PRO Config") -> HelloRsp`
- `DeviceClient.get_config() -> DeviceConfig`
- `DeviceClient.set_config(config: DeviceConfig, apply: bool = True, save: bool = False) -> DeviceConfig`
- `DeviceClient.save_config() -> None`
- `DeviceClient.factory_reset() -> None`
- `DeviceClient.get_status() -> DeviceStatus`
- `SerialTransport(port: str, baudrate: int = 115200, open_timeout: float = 2.0)`

### 3. Contracts

- UI pages call `DeviceClient` operations only through the GUI worker thread; they must not read serial ports or parse frames directly.
- `pc_gui/device/` owns frame encoding, protobuf messages, request IDs, `reply_to` matching, timeouts, and response-code errors.
- `pc_gui/ui/` owns controls, display text, and user interactions only.
- `SerialTransport` imports `serial` lazily so unit tests can run without a connected device.
- PySide6 imports stay in `pc_gui/ui/` and `pc_gui/main.py`; device-layer tests must run without PySide6 installed.
- Developer log events include TX/RX body, `msg_id`, `reply_to`, payload length, frame errors, response codes, and timeouts where available.
- VS Code keeps two GUI task entries: `PC GUI: Run Mock` for no-hardware UI checks and `PC GUI: Run USB CDC ACM` for a real serial port prompt.
- `SerialTransport.open()` must bound Windows/driver-level `serial.Serial(...)` hangs with `open_timeout`; late-open serial handles after timeout must be closed instead of being retained.
- Python protobuf files are generated by `pc_gui/proto/generate_proto.py` using `grpc_tools.protoc`; do not use the system Anaconda `protoc` as the source of truth.
- `DeviceClient.set_config()` must inspect `ConfigSetRsp.code`; non-OK codes are device rejections and must raise `DeviceResponseError` even though the protobuf body is not the generic `Response` message.

### 4. Validation & Error Matrix

- Bad frame magic/version/CRC -> raise `FrameDecodeError`.
- Oversized payload encode -> raise `ValueError`.
- `reply_to` mismatch -> raise `DeviceResponseError(ResponseCode.INTERNAL_ERROR, ...)`.
- Device `Response` with non-OK code -> raise `DeviceResponseError` carrying the numeric code.
- `ConfigSetRsp.code != RESPONSE_CODE_OK` -> raise `DeviceResponseError` carrying the numeric code.
- No response before timeout -> raise `TimeoutError` and log `timeout`.
- Serial read/write before open -> raise `RuntimeError`.
- Serial port constructor blocks longer than `open_timeout` -> raise `RuntimeError` with USB CDC ACM port diagnostics.
- Missing PySide6 -> GUI cannot launch, but `pc_gui/tests` must still run.

### 5. Good / Base / Bad Cases

- Good: RGB page reads cached `DeviceConfig`, edits fields, then submits `DeviceClient.set_config()` through `DeviceWorker`.
- Good: developer log page receives protocol log events through a handler instead of scraping UI state.
- Base: app starts with `MockTransport` when no serial port is provided so the GUI can be developed without hardware.
- Base: VS Code `PC GUI: Run Mock` starts without `--port`; `PC GUI: Run USB CDC ACM` prompts for a COM port and passes `--port`.
- Bad: a button handler calls `SerialTransport.read()` directly.
- Bad: a UI widget manually constructs bytes such as `b"\x55\xaa"` instead of using `FrameCodec`.
- Bad: launching the real-port GUI through a bare Python/Anaconda interpreter or without an open timeout around `serial.Serial(...)`.
- Bad: device-layer unit tests import PySide6 and fail in a headless/toolchain Python environment.

### 6. Tests Required

- Test frame round trip, CRC rejection, partial frame, oversized payload, and stream decoding with noise.
- Test `DeviceConfig` default values and dict/protobuf round trip.
- Test `DeviceClient` with `MockTransport` for hello, get config, set config, save, status, and factory reset.
- Test `DeviceClient.set_config()` raises when mock/device returns a non-OK `ConfigSetRsp.code`.
- Test `SerialTransport.open()` diagnostics for immediate open failures, constructor timeout, and late-open handle cleanup.
- Run tests with plugin autoload disabled if the global Python environment has unrelated pytest plugins:
  `$env:PYTEST_DISABLE_PLUGIN_AUTOLOAD='1'; E:\ncs\toolchains\fd21892d0f\opt\bin\python.exe -m pytest pc_gui\tests`.

### 7. Wrong vs Correct

#### Wrong

```python
def apply_config(self) -> None:
    frame = b"\x55\xaa" + payload
    self.serial.write(frame)
    self.serial.read(64)
```

#### Correct

```python
def apply_config(self) -> None:
    self.window.run_device_task(
        "apply_config",
        lambda client: client.set_config(config, apply=True, save=False),
    )
```
